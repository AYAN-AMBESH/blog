To learn about vulnerabilities and how to find them, one thing I noticed was there wasn't any methodology mentioned on PortSwigger that I could follow. (We are supposed to create our own methodology.)

## What do you think a website is? What about HTTP or HTTPS?

If you learn about networking, a part of computer science, you will get to know about the OSI and TCP/IP models, and you will find protocols in every layer of these models. The Application Layer is the interface through which your software application (web browser) exchanges data via protocols like HTTP or HTTPS.

So, if I were to explain this like ELI5 (Explain Like I'm 5), HTTP/HTTPS are ways through which two computer systems communicate and share data with each other. I won't go into details about networking as this is the part I'm leaving for the reader to find out themselves.

Now about the topic, HTTP/HTTPS works as if you (Bob) are asking your friend (Alice) where a certain book is stored in her house and Bob requires it. So Bob sends a request to Alice with the specifications of the book, like "I want the Real-World Bug Hunting book," and tells Alice that it might be on the third row of the bookshelf in her room on the second floor. After receiving the request, Alice replies with a response that she found the book and gives it to Bob. (If Bob and Alice are good friends, Bob should always return the book back.)

<div style={styles.flow}>
Bob (Client)
      -> Sends HTTP Request
      -> Alice (Server)
      -> Sends HTTP Response
      -> Bob (Client)
</div>

Now you must understand how HTTP really works. Let's get deeper into it.

Since HTTP/HTTPS are stateless protocols, meaning every request is independent of each other, how do you think a real-world application works if it contains multiple users and each user is required to login?

That's where cookies and sessions come into the picture.

Cookies and sessions are used to store the user authentication data so every time a request is made through that user, a cookie or a session ID is sent through that request so that the server can identify which user has requested the data and which user's data should be sent via the response.

## Request & Response

To understand pentesting better, you should first understand how data moves between client and server in every interaction. Whenever you click, submit, or load anything on a website, a request is sent and a response is returned. If you can read this flow clearly, you can map behaviour, trust boundaries, and possible weak points faster during reconnaissance.

HTTP and HTTPS are defined using specific RFC (Request for Comments), i.e. RFC 9110 and RFC 9112 (there are older RFC documents but these mentioned are the latest) for HTTP and RFC 2818 for HTTPS.

Browser
      -> HTTP Request
      -> Web Server
      -> HTTP Response
      -> Browser
</div>

### Request Line

The Request Line contains three important pieces of information.

#### HTTP Method

The HTTP Method defines what action the client wants the server to perform.

- **GET** — Fetch data.
- **POST** — Submit data.
- **PUT** — Update existing data.
- **DELETE** — Delete a specific resource.

#### Target

The Target is the path of the requested resource (URI).

Examples:

```text
/admin
/login
/logout
````

#### HTTP Version

The version of the HTTP protocol being used for the request.

---

### Request Headers

Headers contain metadata about how the request should be handled.

Some common examples include:

* Domain name of the server.
* Information about the browser.
* Content types.
* Cookies.
* Authentication headers.

Think of headers as additional information that helps the server correctly process the request.

---

### Request Body

The Request Body carries the data from the client to the server.

Examples include:

* Login credentials
* Registration forms
* JSON payloads
* XML requests
* File uploads

Not every HTTP request contains a body. GET requests, for example, usually do not.

### Status Line

The Status Line contains:

* HTTP Version
* Status Code

A Status Code is a three-digit number used for classification of the server response.

---

### Response Headers

Response Headers contain metadata detailing how the response should be handled.

Examples include:

* Content-Type
* Cache-Control
* Set-Cookie
* Server
* Content-Length

---

### Response Body

The Response Body contains the actual content that is sent back to the client.

Depending on the application, this may include:

* HTML pages
* JSON responses
* XML data
* Images
* JavaScript
* Downloadable files

## Reconnaissance Phase

The first phase (Reconnaissance) is to map out the web application, which you can also do with the help of tools such as Burp Suite.

<div style={styles.lead}>
      <strong>Core idea:</strong> Before picking any vulnerability, first understand
      how the application behaves. Reconnaissance is about building a complete map
      of the application rather than immediately testing payloads.
</div>

Before picking any vulnerability:

## 1. Map the Attack Surface

<div style={styles.lead}>
      <strong>Core idea:</strong> Every vulnerability begins with an input.
      Before testing anything, identify every location where user-controlled
      data enters the application.
</div>

Map every possible input that the application accepts.

- List every input:
  - URL parameters
  - Query strings
  - Form fields
  - Headers (User-Agent, X-Forwarded-For, etc.)
  - Cookies
  - JSON/XML body fields
  - Multipart file uploads
  - WebSocket messages
  - GraphQL queries

- Map all the API endpoints:

```text
/api
/graphql
/swagger.json
/openapi.json
````

* Identify the tech stack:

  * Use Wappalyzer.
  * Check Response Headers.
  * Look for error messages.
  * Inspect JavaScript source maps.

<div style={styles.flow}>
User Input
      ↓
Application Entry Point
      ↓
Request Processing
      ↓
Response
</div>

## 2. Context Map

<div style={styles.lead}>
      <strong>Core idea:</strong> Finding where your input appears is often more
      valuable than immediately trying payloads. Context determines which
      vulnerability classes are even possible.
</div>

* For each input send a harmless unique string (e.g. `T9a4sl@p`) and find every place it reappears in the response.

* Note what surrounds the string.

This context decides which vulnerability class applies.

* Send a junk probe string with special characters:

```text
<>"';--|&#${}
```

and record:

* Which survive.
* Which get encoded.
* Which get stripped.

<div style={styles.flow}>
Input
      ↓
Reflection
      ↓
Context
      ↓
Potential Vulnerability
</div>

## 3. Determine Response Behaviour

<div style={styles.lead}>
      <strong>Core idea:</strong> Before deciding which vulnerability to test,
      understand how the application responds to different requests.
</div>

* Check the **Content-Type** response header.

If it is not `text/html`, XSS may not fire regardless of payload.

* Note WAF behaviour.

Look for:

* Blocked response patterns.

* Rate limiting headers.

* Test HTTP Method Override.

Example:

```http
X-HTTP-Method-Override: PUT
```

* Only now decide which vulnerability category to test for this input.

Map Inputs
      ↓
Find Reflection
      ↓
Understand Context
      ↓
Observe Response Behaviour
      ↓
Choose Vulnerability Class
</div>

## Mindset Shift

Reconnaissance is not about throwing payloads at every parameter.

It is about understanding how an application behaves before attempting to exploit it.

Beginners often ask:

> Which payload should I use?

Experienced testers usually ask:

> How does this application actually work?

The more complete your understanding of the application, the easier it becomes to identify trust boundaries, hidden functionality, and potential vulnerabilities.

Security is rarely about blindly testing payloads. It is about understanding systems first.
